Reversing Avast Quarantine

Nov 16, 2021 22:03 ¬∑ 1290 words ¬∑ 7 minute read

Intro üîó

In this blog post, I will attempt to describe the Avast Antivirus quarantine file format. We’ll see where the files are stored and how they can be retrieved.

Throughout the process, I will use two files for demonstration purposes. A simple eicar.com and a meterpreter.exe, both put into quarantine.

Part 0 - The overall structure üîó

Avast stores all of its quarantine data inside the folder C:\ProgramData\Avast Software\Avast\chest

Here’s an example of the folder’s content structure

$ ls -la chest/
-rw-r--r--  1 root root 73810 Sep 10 22:46 00000005
-rw-r--r--  1 root root   503 Sep 10 23:35 index.xml

File index.xml is the central place where all the metadata related to quarantine files are stored. Information such as the filename, size, path, detection date etc. Files starting with 0000000X contain raw bytes of the quarantined files. We’ll see later that this is not always the case.

The quarfile starts with 8 ASCII bytes -chest- . Everything afterwards are the raw encrypted bytes of our meterpreter.exe. We can verify this by comparing the length of the file (minus 8 bytes) with the value of the FileSize field in index.xml.

Obtaining the decryption key üîó

Luckily for us, Avast uses a simple XOR cipher. To get the key, we can simply xor the quarantine file 00000005 with its known plaintext form: meterpreter.exe. In doing so, the key is revealed.

Let’s see this in CyberChef.

So far, we only got a partial key. To get the whole key, we need to know its length. Usually, at this point I would try to find some repeatable patterns which would give away the key length.

The key starts with a byte sequence 33b65983. By doing a simple search, we find another occurence at offset 10240, which when divided by two gives us a key length of 5120 bytes.

Upon decrypting the database, we get the following tables

Raw quarantine files are stored as binary blobs in the stream table. The filenames are stored in the vault table. To link them together, we have to join the tables, along with the auxiliary table crossRef.

The following SQL code accomplishes that

SELECT data FROM vault 
INNER JOIN crossRef ON id = crossRef.vaultid 
INNER JOIN stream ON crossRef.streamId = stream.streamId
WHERE filename LIKE "%eicar%"

The result is the original file eicar.com
